# Requirements — `www-site-redesign`

Redesign the coco marketing site (`.www`, a Next.js app deployed at git-co.co) from a
single-purpose "AI commit message" landing page into a dark, terminal-themed product
site that positions coco as a full git toolbelt — individual commands first, the
workstation as the payoff — with an auto-maintained docs system sourced from the
GitHub wiki.

Requirements are written in EARS form. Each is numbered so design properties, tasks,
and tests can trace back to it (e.g. the `mergeManifests` property test header reads
*"Validates: Requirements 6.4, 6.5"*).

## 1. Positioning & hero

- 1.1 THE Hero_Section SHALL lead with the concrete value of individual commands
  (`coco commit`, `coco review`, `coco changelog`, `coco pr create`) before
  introducing the workstation.
- 1.2 THE Hero_Section SHALL present a single primary install affordance
  (`brew install gfargo/tap/coco`) with a one-line copy control.
- 1.3 THE Hero_Section SHALL render a representative motion asset (a commit-flow GIF)
  above the fold without blocking first paint.
- 1.4 THE Website SHALL describe coco as a "git toolbelt / terminal workstation", not
  solely a commit-message generator, in primary copy and metadata.

## 2. Information architecture & navigation

- 2.1 THE Site SHALL expose top-level routes for the home page, the workstation
  showcase, the multi-forge story, the changelog, and the docs.
- 2.2 THE Navigation SHALL remain reachable on mobile via a collapsible control.
- 2.3 WHEN a visitor lands on any route, THE Site SHALL render a consistent
  terminal-themed header and footer.

## 3. Workstation showcase

- 3.1 THE Workstation_Page SHALL present the 16 workstation views with captured
  screenshots sourced from the screenshot pipeline (`/public/screenshots`).
- 3.2 THE Workstation_Page SHALL document chord navigation, the command palette, and
  one-keystroke workflows.
- 3.3 WHEN screenshots are regenerated by `npm run screenshot:sync`, THE
  Workstation_Page SHALL reflect them with no manual file edits.

## 4. Changelog system

- 4.1 THE Changelog_Page SHALL render release notes sourced from the repository's
  `RELEASE_NOTES_*.md` files.
- 4.2 THE Changelog_System SHALL derive each release's version from its filename
  (`RELEASE_NOTES_{major}.{minor}.{patch}.md`) such that constructing a filename from
  a version and parsing it back yields the original version. *(round-trip property)*
- 4.3 THE Changelog_Page SHALL order releases newest-first.

## 5. Theme gallery

- 5.1 THE Themes_Page SHALL present the theme catalog derived from the codebase's
  source of truth (`THEME_PRESET_COLORS`), not a hand-maintained list.
- 5.2 THE Themes_Page SHALL render each theme's captured carousel screenshot.
- 5.3 WHEN a theme is added to `THEME_PRESET_COLORS` and screenshots are synced, THE
  Themes_Page SHALL include it without a manual manifest edit.

## 6. Docs system (wiki integration)

- 6.1 THE Docs_System SHALL scan `.wiki/` at build time so new wiki pages appear
  without manual manifest updates.
- 6.2 THE Docs_System SHALL derive a URL slug for each wiki page from its filename.
- 6.3 THE derived slug SHALL be valid kebab-case: lowercase, alphanumerics and single
  hyphens only, no leading/trailing/doubled hyphens, `.md` stripped. *(slug property)*
- 6.4 THE Docs_System SHALL merge a hand-curated manifest with auto-discovered pages
  such that every manual entry appears exactly once with its fields unchanged, every
  non-overlapping discovered page appears exactly once, and no discovered page whose
  `wikiPath` matches a manual entry appears. *(merge property)*
- 6.5 THE merged result SHALL contain no duplicate `wikiPath` values; its length SHALL
  equal `manual.length + non-overlapping-discovered.length`. *(merge property)*
- 6.6 WHEN a wiki page is requested at `/docs/[slug]`, THE Docs_System SHALL render its
  markdown with the same code/heading styling as the rest of the site.

## 7. Multi-forge story

- 7.1 THE Compare_Page SHALL communicate that coco works across GitHub, GitHub
  Enterprise, GitLab, and Bitbucket from one tool.
- 7.2 THE Site SHALL provide a forge-specific entry (e.g. `/gitlab`) for visitors
  arriving with a non-GitHub mental model.

## 8. Visual design

- 8.1 THE Site SHALL use a dark, terminal-inspired theme (monospace accents, terminal
  palette) consistent with the TUI's aesthetic.
- 8.2 THE Site SHALL render code and command snippets in a styled, copyable form.

## 9. Performance

- 9.1 THE Website SHALL achieve a Lighthouse performance score of 90 or above on
  mobile for the home route.
- 9.2 THE Site SHALL serve optimized image assets (WebP variants where available) and
  avoid layout shift from motion assets.

## 10. Accessibility

- 10.1 THE Site SHALL meet WCAG AA contrast for text against the dark theme.
- 10.2 THE Navigation and interactive controls SHALL be keyboard-operable with visible
  focus states.

## 11. SEO & metadata

- 11.1 THE Site SHALL provide per-route titles, descriptions, and Open Graph metadata.
- 11.2 THE Site SHALL expose a sitemap covering home, workstation, changelog, themes,
  compare, and every docs slug.

## 12. Responsiveness

- 12.1 THE Site SHALL render correctly from a 360px mobile viewport up to wide desktop.
- 12.2 THE motion and screenshot assets SHALL scale within their containers without
  overflow on small viewports.
