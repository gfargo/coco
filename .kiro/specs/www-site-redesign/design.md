# Design — `www-site-redesign`

How the redesigned `.www` site is structured and the correctness properties that
constrain its data layer. The site is a Next.js (App Router) application; the docs and
changelog systems are build-time-static, sourced from the repo's `.wiki/` and
`RELEASE_NOTES_*.md` files so content tracks the codebase with no manual manifest
upkeep.

## Architecture overview

```
.wiki/*.md ─┐                         build time
            ├─ scan ─► discovered-pages.ts ─┐
manifest ───┘  (prebuild.mjs)               ├─ mergeManifests ─► WikiPage[] ─► /docs routes
                                            │
RELEASE_NOTES_*.md ─► parseVersionFromFilename ─► sorted releases ─► /changelog
THEME_PRESET_COLORS (coco src) ─► theme catalog + synced screenshots ─► /docs/themes
```

The data transforms (slug derivation, manifest merge, version parsing) are pure
functions with property-based tests, because they are the parts most likely to break
silently as content scales.

## Route map

| Route             | Source                              | Requirements        |
|-------------------|-------------------------------------|---------------------|
| `/` (`_home`)     | Hero + sections (commands → workstation) | 1.x, 2.x, 8.x   |
| `/workstation`    | Screenshot pipeline assets          | 3.x                 |
| `/gitlab`         | Forge-specific landing              | 7.2                 |
| `/compare`        | Multi-forge comparison              | 7.1                 |
| `/changelog`      | `RELEASE_NOTES_*.md`                | 4.x                 |
| `/docs`           | Merged wiki manifest (index)        | 6.1, 6.4, 6.5       |
| `/docs/[slug]`    | Per-page wiki markdown              | 6.2, 6.3, 6.6       |
| `/docs/themes`    | `THEME_PRESET_COLORS`-derived gallery | 5.x               |

`_home/sections/` decomposes the landing page into ordered sections so the hero leads
with command value (1.1) and the workstation appears as the payoff lower on the page.

## Data model

The wiki page record (`.www/src/lib/wiki/wiki-manifest.ts`):

```ts
type WikiPage = {
  slug: string            // URL slug, kebab-case
  title: string           // display title
  wikiPath: string        // source filename stem in .wiki/ (the merge key)
  category: string        // grouping in the docs nav
  order: number           // sort within category
  description?: string    // optional summary
  isAutoDiscovered?: true // set only on pages found by the build-time scan
}
```

`wikiPath` is the identity key across the manual manifest and the auto-discovered set —
the merge dedupes on it.

## Component / module interfaces

Pure data functions (each with co-located example tests and a `*.property.test.ts`):

```ts
// slug-utils.ts — derive a URL slug from a wiki filename
deriveSlugFromFilename(filename: string): string

// merge-manifests.ts — manual entries win; non-overlapping discovered pages appended
mergeManifests(manual: WikiPage[], discovered: WikiPage[]): WikiPage[]

// changelog/parse-version.ts — extract a semver from RELEASE_NOTES_x.y.z.md
parseVersionFromFilename(filename: string): string | null
```

Build-time generation:

- `discovered-pages.ts` is **auto-generated by `prebuild.mjs`** (header:
  `// This file is auto-generated by prebuild.mjs — do not edit manually.`). The
  prebuild scans `.wiki/` and emits the discovered `WikiPage[]`; at request/build time
  `mergeManifests(manualManifest, discoveredPages)` produces the final nav and route
  set. New wiki pages therefore appear with no manifest edit (6.1).

## Correctness properties

These constrain the data layer regardless of input. Properties 1–3 are enforced by
`fast-check` property-based tests (100 runs each); Property 4 is a build-time invariant
covered by example tests over the real catalog.

**Property 1 — Slug derivation produces valid kebab-case.** *(Requirements 6.2, 6.3)*
For any wiki filename matching `[A-Z][A-Za-z0-9&-]+\.md`, `deriveSlugFromFilename`
returns a string that is lowercase, contains only `[a-z0-9-]`, has no leading/trailing
or doubled hyphens, and has `.md` stripped.
→ `.www/src/lib/wiki/slug-utils.property.test.ts`

**Property 2 — Manifest merge preserves manual entries and defaults the rest.**
*(Requirements 6.4, 6.5)* For any manual and discovered `WikiPage[]` (each internally
unique by `wikiPath`): (a) every manual entry appears exactly once, fields unchanged;
(b) every non-overlapping discovered page appears exactly once; (c) no discovered page
whose `wikiPath` matches a manual entry appears; (d) the result length equals
`manual.length + non-overlapping-discovered.length`.
→ `.www/src/lib/wiki/merge-manifests.property.test.ts`

**Property 3 — Changelog version extraction round-trips.** *(Requirement 4.2)* For any
semver `M.m.p`, `parseVersionFromFilename("RELEASE_NOTES_${version}.md")` returns the
original `version`.
→ `.www/src/lib/changelog/parse-version.property.test.ts`

**Property 4 — Theme catalog is fully derived.** *(Requirements 5.1, 5.3)* Every preset
in coco's `THEME_PRESET_COLORS` maps to exactly one gallery entry, and the gallery
contains no entries without a backing preset — so adding a theme upstream is the only
edit needed. Covered by example tests over the live catalog.

## Performance design (Requirement 9)

- Static generation for all routes; no per-request data fetching.
- Screenshot assets are pre-optimized by the pipeline (pngquant + WebP) and served from
  `/public/screenshots`; motion assets reserve their box to avoid layout shift (9.2).
- Target Lighthouse mobile ≥ 90 on `/` (9.1), validated in CI/preview.

## Out of scope

- The wiki content itself (authored in the `.wiki` repo).
- The screenshot capture mechanism (owned by `bin/screenshot/`, consumed here as assets).
- The coco CLI/TUI (a separate repository concern).
